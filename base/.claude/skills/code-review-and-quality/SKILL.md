# 코드 리뷰와 품질

## 개요

품질 게이트가 있는 다차원 코드 리뷰. 모든 변경은 머지 전 리뷰한다 — 예외 없음. 리뷰는 다섯 축을 다룬다: **correctness(정확성), readability(가독성), architecture(아키텍처), security(보안), performance(성능).**

**승인 기준:** 완벽하지 않더라도 전체 코드 건강을 분명히 개선하면 승인한다. 완벽한 코드는 없다 — 목표는 지속적 개선이다. "내가 짰을 방식과 다르다"는 이유로 막지 않는다. 코드베이스를 개선하고 프로젝트 컨벤션을 따르면 승인한다.

> 이 스킬과 `/cross-check`의 차이: **이것은 Claude(메인 에이전트)가 5축으로 직접 수행하는 구조화된 자체 리뷰**다. `/cross-check`는 **Codex(다른 모델)에게 read-only로 맡기는 외부 교차검증**으로, 서로 다른 추론 경로의 "제2의 눈"이다. 둘은 보완 관계 — 비자명한 변경은 자체 5축 리뷰 후 `/cross-check`까지 거치면 사각지대가 크게 줄어든다.

## 언제 쓰나

- PR/변경을 머지하기 전
- 기능 구현을 마친 뒤
- 다른 에이전트·모델이 만든 코드를 평가해야 할 때
- 기존 코드를 리팩터링할 때
- 버그 수정 후 (수정과 회귀 테스트 모두 리뷰)

### When NOT to use

- 변경이 없는데 형식상 리뷰를 흉내내는 것 (리뷰할 diff가 비면 알리고 중단)
- 오타 한 줄·자동 포맷 같은 자명한 변경 (5축은 과잉 — reviewer 재량으로 생략)
- 코드 정확성 버그 헌팅이 주목적이면 `/cross-check`(Codex)나 `/code-review`가 더 적합 — 이 스킬은 다축 품질 평가에 초점

## 핵심 프로세스: 5축 리뷰

모든 리뷰는 다섯 차원에서 코드를 평가한다.

### 1. Correctness (정확성)

코드가 주장하는 일을 하는가?

- 스펙/작업 요구사항에 부합하는가?
- 엣지 케이스(null, 빈 값, 경계값)가 처리되는가?
- 에러 경로가 처리되는가 (happy path만이 아니라)?
- 모든 테스트를 통과하는가? 테스트가 옳은 것을 검증하는가?
- off-by-one, 경쟁 조건, 상태 불일치가 있는가?
- ML/DL: shape/dtype/device 일치, seed/재현성, loss/metric 정의 일치, train/eval split 무결성

### 2. Readability & Simplicity (가독성·단순성)

다른 엔지니어(또는 에이전트)가 설명 없이 이해할 수 있는가?

- 이름이 서술적이고 프로젝트 컨벤션과 일관적인가? (`temp`, `data`, `result` 남발 금지)
- 제어 흐름이 직관적인가? (중첩 삼항, 깊은 콜백 회피)
- 코드가 논리적으로 조직되었는가?
- 단순화해야 할 "영리한" 트릭은 없는가?
- **더 적은 줄로 가능한가?** (100줄로 될 것을 1000줄로 짜면 실패)
- **추상화가 복잡성을 정당화하는가?** (세 번째 사용처가 나오기 전에 일반화하지 않는다)
- 죽은 코드(no-op 변수, 하위호환 shim, `# removed` 주석)는 없는가?
- (CLAUDE.md "Simplicity First", "Demand Elegance"와 일치.)

### 3. Architecture (아키텍처)

변경이 시스템 설계에 맞는가?

- 기존 패턴을 따르는가, 새 패턴을 도입하는가? 새것이면 정당한가?
- 깔끔한 모듈 경계를 유지하는가?
- 공유해야 할 코드 중복은 없는가?
- 의존성이 올바른 방향으로 흐르는가 (순환 의존성 없음)?
- 추상화 수준이 적절한가 (과잉 설계도, 과결합도 아닌)?

### 4. Security (보안)

상세 가이드는 `security-and-hardening` 스킬과 `harness/references/security-checklist.md`를 본다. 변경이 취약점을 들이는가?

- 사용자 입력이 검증·sanitize되는가?
- 시크릿이 코드·로그·버전 관리에서 배제되는가?
- 필요한 곳에 인증/인가가 확인되는가?
- SQL 쿼리가 파라미터화되는가 (문자열 연결 없음)?
- 출력이 XSS 방지를 위해 인코딩되는가?
- 외부 출처(API, 로그, 사용자 콘텐츠, config) 데이터가 신뢰 불가로 취급되는가?
- ML/DL: 외부 pickle/torch.load 역직렬화, 데이터 유출 경계

### 5. Performance (성능)

변경이 성능 문제를 들이는가?

- N+1 쿼리 패턴은 없는가?
- 무한 루프나 제약 없는 데이터 fetch는 없는가?
- 비동기여야 할 동기 연산은 없는가?
- 리스트 엔드포인트에 페이지네이션이 누락되지 않았는가?
- hot path에서 큰 객체를 만들지 않는가?
- ML/DL: 불필요한 GPU↔CPU 복사, 매 step의 동기화, 비효율적 데이터로더

## 변경 크기 (Change Sizing)

작고 집중된 변경이 리뷰하기 쉽고, 빠르게 머지되며, 안전하게 배포된다.

```
~100줄 변경   → 좋음. 한 자리에서 리뷰 가능.
~300줄 변경   → 단일 논리 변경이면 수용 가능.
~1000줄 변경  → 너무 큼. 쪼개라.
```

**"하나의 변경"이란:** 한 가지를 다루고, 관련 테스트를 포함하며, 제출 후에도 시스템이 동작하는 자기완결적 수정. 기능의 한 부분 — 전체 기능이 아니다.

**너무 클 때 쪼개는 전략:**

| 전략 | 방법 | 시점 |
|---|---|---|
| **Stack** | 작은 변경을 제출하고 그 위에 다음 변경 시작 | 순차 의존성 |
| **파일 그룹별** | 다른 리뷰어가 필요한 그룹을 분리 | 횡단 관심사 |
| **Horizontal** | 공유 코드/스텁 먼저, 그다음 소비자 | 계층 구조 |
| **Vertical** | 기능을 더 작은 풀스택 슬라이스로 | 기능 작업 |

**큰 변경이 허용될 때:** 파일 전체 삭제, 리뷰어가 의도만 확인하면 되는 자동 리팩터링.

**리팩터링과 기능 작업을 분리하라.** 기존 코드를 리팩터하면서 새 동작을 추가하는 변경은 두 개의 변경이다 — 따로 제출한다.

## 변경 설명 (Change Descriptions)

모든 변경에는 버전 관리 히스토리에서 단독으로 읽히는 설명이 필요하다.

- **첫 줄:** 짧고 명령형이며 독립적. "FizzBuzz RPC 삭제" (not "Deleting..."). 히스토리 검색자가 diff 없이 이해할 만큼 정보를 담는다.
- **본문:** 무엇이 왜 바뀌는지. 코드에 안 보이는 맥락·결정·근거. 버그 번호, 벤치마크, 설계 문서를 링크.
- **안티패턴:** "Fix bug", "Fix build", "Phase 1", "Add convenience functions". (우리 `/commit`은 Conventional Commits를 따른다.)

## 리뷰 절차

### 1단계: 맥락 이해

코드를 보기 전에 의도를 이해한다 — 이 변경이 무엇을 달성하려 하는가? 어떤 스펙/작업을 구현하는가? 기대되는 동작 변화는?

### 2단계: 테스트 먼저 리뷰

테스트는 의도와 커버리지를 드러낸다 — 변경에 테스트가 있는가? 구현 세부가 아니라 동작을 테스트하는가? 엣지 케이스가 커버되는가? 이름이 서술적인가? 코드가 바뀌면 회귀를 잡는가?

### 3단계: 구현 리뷰

변경된 각 파일을 5축(correctness/readability/architecture/security/performance)으로 훑는다.

### 4단계: 발견 분류 (심각도 라벨)

모든 코멘트에 심각도를 붙여 무엇이 필수이고 선택인지 작성자가 알게 한다:

| 라벨 | 의미 | 작성자 조치 |
|---|---|---|
| **Critical:** | 머지 차단 | 보안 취약점, 데이터 손실, 깨진 기능 — 머지 전 반드시 해결 |
| **Important:** | 중요 | 머지 전 해결 또는 정당화와 함께 명시적 연기 |
| *(라벨 없음)* | 필요 변경 | 머지 전 처리 |
| **Optional:** / **Consider:** | 제안 | 고려할 가치 있으나 필수 아님 |
| **Nit:** | 사소·선택 | 작성자가 무시 가능 — 포맷, 스타일 취향 |
| **FYI** | 정보성 | 조치 불필요 — 향후 참고 맥락 |

심각도 라벨이 없으면 작성자가 모든 피드백을 필수로 받아들여 시간을 낭비한다. (`/cross-check`도 `[CRITICAL|HIGH|MEDIUM|LOW]`로 분류 — 일관성 유지.)

### 5단계: 검증의 검증

작성자의 검증 스토리를 확인한다 — 어떤 테스트를 돌렸나? 빌드는 통과했나? 수동 테스트했나? UI 변경의 스크린샷이 있나? before/after 비교가 있나? (CLAUDE.md "Verification Before Done" 적용.)

## 패턴

### 멀티모델 리뷰 패턴

서로 다른 모델로 서로 다른 리뷰 관점을 얻는다. 우리 환경에서는 **Claude가 코드를 짜고 5축 자체 리뷰**한 뒤, 비자명하면 **`/cross-check`로 Codex(다른 모델)**가 read-only로 버그·사각지대를 다시 본다. Codex 출력은 결론이 아니라 입력 — Claude가 각 finding을 ✅동의/⚠️부분동의/❌오탐으로 판정한다 (CLAUDE.md "교차검증 루프").

```
Claude가 코드 작성
    │
    ▼
Claude 5축 자체 리뷰 (이 스킬)
    │
    ▼ (비자명한 변경이면)
/cross-check → Codex가 read-only 교차검증
    │
    ▼
Claude가 finding 판정 + 동의 항목만 수정
    │
    ▼
사람이 최종 결정 (harness/decision-log.md)
```

### 죽은 코드 위생 (Dead Code Hygiene)

리팩터링·구현 변경 후 고아 코드를 확인한다: (1) 이제 도달 불가/미사용 코드 식별, (2) 명시적으로 나열, (3) **삭제 전 묻기** — "이 미사용 요소들을 지울까요: [목록]?". 죽은 코드를 방치하면 미래 독자·에이전트를 혼란시킨다. 단, 확신 없는 것을 조용히 지우지 않는다.

```
DEAD CODE IDENTIFIED:
- format_legacy_date() in src/utils/date.py — format_date()로 대체됨
- LEGACY_API_URL in src/config.py — 남은 참조 없음
→ 안전하게 제거할까요?
```

### 리뷰 속도

느린 리뷰는 팀 전체를 막는다. 리뷰로 컨텍스트 전환하는 비용 < 남에게 부과하는 대기 비용. 영업일 1일 내 응답(최대치, 목표 아님), 빠른 개별 응답을 빠른 최종 승인보다 우선, 큰 변경은 쪼개달라고 요청.

### 이견 처리 (Handling Disagreements)

리뷰 분쟁 해결 위계: (1) 기술적 사실·데이터 > 의견·취향, (2) 스타일 가이드가 스타일의 절대 권위, (3) 소프트웨어 설계는 개인 취향이 아닌 공학 원칙으로, (4) 전체 건강을 해치지 않으면 코드베이스 일관성 수용. **"나중에 정리할게"를 받아들이지 마라** — 연기된 정리는 거의 일어나지 않는다 (CLAUDE.md "클린 상태로 종료").

### 리뷰의 정직성 (Honesty in Review)

- **고무도장 찍지 마라.** 리뷰 증거 없는 "LGTM"은 아무도 돕지 않는다.
- **실제 문제를 누그러뜨리지 마라.** 프로덕션을 칠 버그를 "사소한 우려일 수도"라 하는 건 부정직하다.
- **가능하면 정량화하라.** "이 N+1은 항목당 ~50ms 추가"가 "느릴 수 있다"보다 낫다.
- **명백한 문제가 있는 접근에 반박하라.** 아첨은 리뷰의 실패 모드다 (CLAUDE.md "No Laziness").
- **번복을 우아하게 수용하라.** 작성자가 충분한 맥락으로 동의하지 않으면 그 판단을 존중한다. 사람이 아니라 코드를 비평한다.

### 의존성 규율 (Dependency Discipline)

의존성 추가 전: (1) 기존 스택으로 풀 수 있나? (대개 가능), (2) 크기는?, (3) 활발히 유지되나?, (4) 알려진 취약점은? (`security-and-hardening` 참조), (5) 라이선스 호환? **규칙: 표준 라이브러리·기존 유틸을 새 의존성보다 우선.** 모든 의존성은 부채다.

## Common Rationalizations

| 변명 | 반박 |
|---|---|
| "돌아가니까 충분해" | 읽기 어렵거나, 안전하지 않거나, 아키텍처가 틀린 작동 코드는 복리로 불어나는 부채다. |
| "내가 짰으니 맞는 걸 알아" | 작성자는 자기 가정에 눈멀어 있다. 모든 변경은 또 다른 시선의 덕을 본다. |
| "나중에 정리할게" | 나중은 안 온다. 리뷰가 품질 게이트다 — 머지 전에 정리하라. |
| "AI가 짠 코드니 괜찮겠지" | AI 코드는 덜이 아니라 더 검토가 필요하다. 틀려도 자신 있고 그럴듯하다. |
| "테스트가 통과하니 좋아" | 테스트는 필요조건이지 충분조건이 아니다. 아키텍처·보안·가독성 문제는 못 잡는다. |

## Red Flags

- 리뷰 없이 머지된 PR
- 테스트 통과만 확인하고 다른 축을 무시한 리뷰
- 실제 리뷰 증거 없는 "LGTM"
- 보안 민감 변경에 보안 중심 리뷰 부재 (→ `/cross-check` 권장)
- "제대로 리뷰하기엔 너무 큰" 대형 PR (쪼개라)
- 버그 수정 PR에 회귀 테스트 부재
- 심각도 라벨 없는 리뷰 코멘트 — 필수/선택 구분 불가
- "나중에 고칠게" 수용 — 절대 안 일어난다
