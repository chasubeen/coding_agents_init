# 코드 리뷰 체크리스트

머지 전 모든 변경을 5축(정확성·가독성·아키텍처·보안·성능)으로 점검하고, 심각도 라벨로 피드백을 분류하기 위한 체크리스트.

관련 스킬: code-review-and-quality, /cross-check, /review, /commit

## 승인 기준

완벽한 코드는 없다. 변경이 **전반적인 코드 건강을 분명히 개선**하면, 내가 짰을 방식과 다르더라도 승인한다. 프로젝트 컨벤션을 따르고 코드베이스를 개선한다면 통과시킨다.

## 5축 리뷰

### 1. 정확성 (Correctness)

- [ ] 스펙/작업 요구사항과 일치하는가
- [ ] 엣지케이스를 처리하는가 (null, 빈 값, 경계값)
- [ ] 에러 경로를 처리하는가 (해피패스만이 아니라)
- [ ] 테스트를 통과하고, 그 테스트가 올바른 것을 검증하는가
- [ ] off-by-one, 레이스 컨디션, 상태 불일치가 없는가

### 2. 가독성 & 단순성 (Readability)

- [ ] 이름이 서술적이고 컨벤션과 일치하는가 (`temp`, `data`, `result` 단독 사용 지양)
- [ ] 제어 흐름이 직관적인가 (중첩 삼항·깊은 콜백 지양)
- [ ] 코드가 논리적으로 조직되었는가
- [ ] 더 적은 줄로 가능한가 (100줄로 될 일을 1000줄로 짜면 실패)
- [ ] 추상화가 복잡성 비용을 정당화하는가 (세 번째 사용 전엔 일반화하지 않기)
- [ ] 죽은 코드가 없는가 (`_unused` 변수, 호환용 shim, `// removed` 주석)

### 3. 아키텍처 (Architecture)

- [ ] 기존 패턴을 따르는가 (새 패턴이면 정당한가)
- [ ] 모듈 경계가 깨끗한가
- [ ] 공유되어야 할 중복 코드가 없는가
- [ ] 의존성 방향이 올바른가 (순환 의존 없음)
- [ ] 추상화 수준이 적절한가 (과설계도, 과결합도 아님)

### 4. 보안 (Security)

> 상세는 `references/security-checklist.md` 참조.

- [ ] 사용자 입력을 경계에서 검증·정제하는가
- [ ] 시크릿이 코드·로그·버전관리에서 빠져 있는가
- [ ] 필요한 곳에 인증/인가가 있는가
- [ ] SQL 쿼리가 파라미터화되어 있는가
- [ ] 출력이 XSS 방지를 위해 인코딩되는가
- [ ] 외부 출처 데이터(API, 로그, 사용자 콘텐츠, 설정 파일)를 신뢰 불가로 다루는가
- [ ] (ML) `pickle`/`torch.load` 등 신뢰 불가 역직렬화가 없는가

### 5. 성능 (Performance)

> 상세는 `references/performance-checklist.md` 참조.

- [ ] N+1 쿼리 패턴이 없는가
- [ ] 무제한 루프·무제한 데이터 페치가 없는가
- [ ] 비동기여야 할 동기 작업이 없는가
- [ ] UI 컴포넌트의 불필요한 리렌더가 없는가
- [ ] 리스트 엔드포인트에 페이지네이션이 있는가
- [ ] 핫패스에서 거대 객체를 생성하지 않는가

## 변경 크기 (Change Sizing)

작고 집중된 변경이 리뷰하기 쉽고, 머지가 빠르고, 배포가 안전하다.

```
~100줄 변경    → 좋음. 한 번에 리뷰 가능.
~300줄 변경    → 단일 논리적 변경이면 허용.
~1000줄 변경   → 너무 큼. 쪼갤 것.
```

**리팩터링과 기능 작업을 분리한다.** 기존 코드를 리팩터링하면서 새 동작을 추가하는 변경은 두 개의 변경이다 — 따로 제출한다.

**큰 변경이 허용되는 경우:** 파일 전체 삭제, 자동 리팩터링처럼 리뷰어가 모든 줄이 아니라 의도만 확인하면 되는 경우.

**쪼개는 전략:**

| 전략 | 방법 | 적용 상황 |
|------|------|----------|
| Stack | 작은 변경 제출 후, 그 위에서 다음 변경 시작 | 순차 의존성 |
| 파일 그룹별 | 다른 리뷰어가 필요한 그룹을 분리 | 횡단 관심사 |
| Horizontal | 공유 코드/스텁 먼저, 소비자 나중 | 계층형 아키텍처 |
| Vertical | 풀스택 슬라이스로 작게 분할 | 기능 작업 |

## 심각도 라벨

모든 코멘트에 심각도를 붙여, 작성자가 필수와 선택을 구분하게 한다.

| 접두어 | 의미 | 작성자 조치 |
|--------|------|-------------|
| **Critical:** | 머지 차단 | 보안 취약점, 데이터 손실, 기능 파손 — 반드시 수정 |
| **Important:** / *(접두어 없음)* | 필수 변경 | 머지 전 처리 |
| **Optional:** / **Consider:** | 제안 | 고려할 만하나 필수 아님 |
| **Nit:** | 사소함 | 무시 가능 — 포매팅, 스타일 취향 |
| **FYI** | 정보성 | 조치 불필요 — 향후 참고용 |

라벨이 없으면 작성자가 모든 피드백을 필수로 오해해 선택 사항에 시간을 낭비한다.

## 리뷰 프로세스

1. **맥락 이해** — 이 변경이 무엇을, 왜 하려는지 파악
2. **테스트 먼저** — 테스트가 동작(구현 아님)을 검증하는가, 엣지케이스를 덮는가, 회귀를 잡는가
3. **구현 리뷰** — 파일별로 5축을 적용
4. **findings 분류** — 위 심각도 라벨 부여
5. **검증의 검증** — 어떤 테스트를 돌렸나, 빌드 통과했나, 수동 검증/스크린샷이 있나

## 정직한 리뷰

- **러버스탬프 금지** — 리뷰 흔적 없는 "LGTM"은 누구에게도 도움이 안 된다
- **실제 문제를 누그러뜨리지 말 것** — 운영에서 터질 버그를 "사소한 우려"라 하지 않는다
- **문제를 정량화** — "이 N+1 쿼리는 항목당 ~50ms 추가"가 "느릴 수 있음"보다 낫다
- **명백한 문제는 직접 반박** — 리뷰에서 아첨은 실패 모드다. 문제가 있으면 직설적으로 말하고 대안을 제시한다
- **재정의는 우아하게 수용** — 작성자가 전체 맥락을 갖고 동의하지 않으면 그 판단을 존중. 사람이 아니라 코드를 비평한다
- **"나중에 정리"를 받아들이지 말 것** — 나중은 오지 않는다. 머지 전 정리를 요구한다

## 멀티모델 리뷰 (우리 맥락)

Claude(메인)와 Codex(보조)는 서로 다른 사각지대를 가진다. 비자명한 변경은 `/cross-check`로 Codex 교차검증을 제안한다.

```
Claude가 코드 작성
    │
    ▼
Codex가 정확성·보안·엣지케이스를 read-only로 독립 검토 (/cross-check)
    │
    ▼
Claude가 각 finding을 검증 (✅동의 / ⚠️부분동의 / ❌오탐) 후 수정
    │
    ▼
사람이 최종 판단
```

> Codex 출력은 결론이 아니라 입력이다. 판단을 외주 주는 것이 아니라 증거를 더 모으는 것.

## 검증

- [ ] 모든 Critical 이슈 해결
- [ ] 모든 Important 이슈 해결 또는 정당한 사유로 명시적 보류
- [ ] 테스트 통과
- [ ] 빌드 성공
- [ ] 검증 스토리 문서화 (무엇을 바꿨고 어떻게 검증했는지)

## 흔한 자기합리화

| 합리화 | 현실 |
|---|---|
| "동작하니 충분해" | 읽기 어렵고 안전하지 않은 코드는 복리로 불어나는 부채다 |
| "내가 짰으니 맞다" | 작성자는 자기 가정에 눈멀어 있다 — 제2의 눈이 필요하다 |
| "나중에 정리하자" | 나중은 오지 않는다. 리뷰가 품질 게이트다 |
| "AI가 짠 코드니 괜찮겠지" | AI 코드는 더 적은 게 아니라 더 많은 검증이 필요하다 |
| "테스트 통과했으니 좋다" | 테스트는 필요조건이지 충분조건이 아니다 |
