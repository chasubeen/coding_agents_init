# 오케스트레이션 패턴

멀티에이전트 오케스트레이션의 규율과 안티패턴. 새 조정형 슬래시 커맨드를 추가하기 전에 읽는다. 우리 맥락에서는 `/cross-check`(Codex 교차검증)와 `/orchestrate`(Codex 병렬 워커)에 이 규율을 적용한다.

관련 스킬: /orchestrate, /cross-check

## 지배 규율

> **사용자(또는 슬래시 커맨드)가 오케스트레이터다. 에이전트는 다른 에이전트를 호출하지 않는다.**

세 가지 핵심 규칙:

- **깊이 ≤ 1** — 조정 깊이는 최대 1단계 (슬래시 커맨드 → 워커들). 머지는 메인 에이전트(Claude) 컨텍스트에서 일어난다
- **fan-out + merge만** — 독립 작업을 병렬로 펼치고(fan-out), 결과를 한 번 모은다(merge). 그 외 토폴로지는 도입하지 않는다
- **router / meta-orchestrator 금지** — "어떤 에이전트를 부를지 결정하는 것"만 하는 계층을 만들지 않는다

## 권장 패턴

### 1. 직접 호출 (오케스트레이션 없음)

단일 관점, 단일 산출물. 기본값이자 가장 저렴한 선택.

```
사용자 → Claude → 보고 → 사용자
```

**사용 시점:** 작업이 하나의 산출물에 대한 하나의 관점이고, 한 문장으로 설명되는 경우. 모든 조정형 패턴은 이 비용과 비교한다.

### 2. 병렬 fan-out + merge

여러 독립 워커가 같은(또는 분할된) 입력에 동시에 작동하고, 각자 독립 보고를 낸다. 메인 에이전트 컨텍스트의 merge 단계가 이를 하나의 결정으로 종합한다.

```
                    ┌─→ 워커 A ─┐
fan out  ───────────┼─→ 워커 B ─┤→ merge → 결정
                    └─→ 워커 C ─┘
```

**채택 전 검증 체크리스트:**
- [ ] 모든 워커를 순서 의존 없이 동시에 돌릴 수 있는가
- [ ] 각 워커가 *다른 종류*의 결과를 내는가 (같은 결과의 다른 각도가 아님)
- [ ] merge 단계가 메인 에이전트의 남은 컨텍스트에 들어가는가
- [ ] 병렬화로 체감될 만큼 대기 시간이 긴가

하나라도 "아니오"면 직접 호출로 폴백한다.

### 3. 사용자 주도 순차 슬래시 커맨드

오케스트레이터 에이전트 없이, 사용자가 정해진 순서로 슬래시 커맨드를 실행한다. 사용자가 곧 오케스트레이터다.

```
사용자 실행:  /feature → /todo → /quality-gate → /cross-check → /commit
```

**사용 시점:** 단계 간 의존성이 있고, 단계 사이 사람의 판단이 가치를 더할 때. LLM "라이프사이클 오케스트레이터"는 단계 간 요약으로 뉘앙스를 잃고, 사람 체크포인트를 건너뛰고, 토큰 비용을 두 배로 만든다.

### 4. 리서치 격리 (컨텍스트 보존)

많은 자료를 읽어야 하지만 그 내용이 메인 컨텍스트를 오염시키면 안 될 때, 다이제스트만 반환하는 리서치 서브에이전트를 띄운다. Claude Code에서는 내장 `Explore` 서브에이전트를 우선 사용한다.

```
메인 → Explore (50개 파일 읽음) → 다이제스트 → 메인 계속
```

## 우리 맥락 매핑: `/cross-check`와 `/orchestrate`

이 저장소의 Claude(메인) + Codex(보조) 구성에서, 위 규율이 두 슬래시 커맨드에 어떻게 적용되는지.

### `/cross-check` — Codex 교차검증

**무엇:** 비자명한 변경 후, Claude가 놓쳤을 버그·엣지케이스·보안 구멍과 설계 대안을 Codex에게 read-only로 검토받는다.

**어느 패턴인가:** 워커 1개짜리 **fan-out + merge**. Codex가 독립 보고를 내고, **merge는 Claude 컨텍스트에서** 일어난다.

```
Claude 작업 → Codex(read-only 검토) → Claude가 각 finding 검증·종합 → 사용자
```

**규율 적용:**
- **깊이 1** — Claude → Codex 한 단계. Codex는 또 다른 에이전트를 부르지 않는다
- **Codex는 읽고 비평만 한다** — 수정은 Claude의 몫. 교차검증은 판단의 외주가 아니라 증거 수집이다
- **merge는 Claude가** — Codex 출력은 결론이 아니라 입력이다. Claude가 각 finding을 ✅동의 / ⚠️부분동의 / ❌오탐으로 검증한다
- **router 금지** — Codex에게 "다음에 무엇을 할지" 결정을 위임하지 않는다

### `/orchestrate` — Codex 병렬 워커

**무엇:** 3개 이상 독립 모듈/파일에 걸친 대규모 개발 작업을, 여러 Codex CLI 에이전트에 병렬 분배해 구현한다.

**어느 패턴인가:** 다수 워커의 **fan-out + merge**. Claude가 작업을 독립 단위로 분해(fan-out), 각 Codex 워커가 서로 다른 파일을 구현, 브랜치 머지(merge).

```
Claude(분해) ─┬─→ Codex agent-01 (파일 그룹 A) ─┐
              ├─→ Codex agent-02 (파일 그룹 B) ─┤→ 머지 → 통합
              └─→ Codex agent-03 (파일 그룹 C) ─┘
```

**규율 적용:**
- **깊이 1** — Claude → Codex 워커들 한 단계. 워커는 하위 워커를 스폰하지 않는다 (meta-orchestrator 금지)
- **작업은 진짜 독립이어야** — 각 워커가 다른 파일을 수정 (파일 충돌 방지). 공유 인터페이스가 필요하면 인터페이스 정의를 첫 워커에 먼저 할당하고 나머지를 의존(deps)으로 순차화
- **merge는 명시적 단계로** — 브랜치 머지/통합을 메인이 책임진다. 워커끼리 서로 호출하거나 조율하지 않는다
- **채택 전 위 검증 체크리스트 통과** — 독립성·다른 결과·merge 적합성·대기 시간. 통과 못 하면 직접 구현 또는 순차 슬래시 커맨드로 폴백

### `/cross-check` vs `/orchestrate` 구분

| | `/cross-check` | `/orchestrate` |
|--|----------------|-----------------|
| Codex 권한 | read-only (검토만) | 쓰기 (구현) |
| 워커 수 | 1 | N (독립 작업 단위) |
| 산출물 | finding 보고 → Claude가 종합 | 코드 변경 → 브랜치 머지 |
| 적합 시점 | 알려진 변경/계획의 검증 | 3+ 독립 파일의 대규모 구현 |

## 패턴 어휘 카탈로그 (6종) — 우리 규율과의 정합성

업계에서 자주 쓰는 6가지 멀티에이전트 패턴이다. 우리는 이 *어휘*를 빌리되, "깊이 ≤1 · router 금지" 규율과 충돌하는 것은 의도적으로 회피하거나 평탄화한다.

| 패턴 | 정의 | 우리 적용 |
|------|------|-----------|
| **Pipeline** | 순차 의존 A→B→C | ✅ **사용자 주도 순차 슬래시 커맨드**(패턴 3)로 구현. 자동 체이닝 대신 사람 체크포인트 유지 |
| **Fan-out / Fan-in** | 병렬 처리 후 통합 | ✅ **패턴 2**. `/orchestrate`(Codex N워커), `/cross-check`(워커 1) |
| **Producer-Reviewer** (generate-verify) | 생성↔검증 쌍, 재시도 2~3회 | ✅ `/cross-check`(Codex 검토), planner/builder/**reviewer** 프로토콜. 무한 루프 방지로 재시도 상한 |
| **Expert pool** (router) | 라우터가 상황별 전문가 선택 | ⚠️ **자동 라우터 금지**. 대신 사용자/Claude가 `AGENTS.md`의 Intent→Skill 매핑으로 직접 선택 |
| **Supervisor** | 중앙 에이전트가 동적 분배·모니터링 | ⚠️ 분배는 **Claude 메인 컨텍스트**가 직접(깊이 1 유지). 별도 supervisor *에이전트*를 두지 않는다 |
| **Hierarchical** | 상위→하위 재귀 위임 | ❌ 깊이 ≤1 위반. **평탄화**한다 (1단계만; 더 필요하면 사용자 주도 2nd 패스) |

복합 패턴(예: fan-out + producer-reviewer)은 가능하나, **전체 조정 깊이는 여전히 ≤1**이어야 한다.

## 실행 모드 선택 — Codex 워커 vs Claude 서브에이전트

2개 이상의 작업 주체가 필요할 때, 우리 환경의 선택지:

```
2개 이상 주체가 필요한가?
├─ 아니오 → Claude 직접 (단일 에이전트)
└─ 예 → 어떤 성격인가?
        ├─ 독립 검토/다른 관점(read-only)      → Codex /cross-check
        ├─ 독립 파일 병렬 구현(write)            → Codex /orchestrate (워커 N)
        ├─ 컨텍스트 오염 없이 대량 읽기          → Claude 내장 Explore 서브에이전트
        └─ 분석/계획만                           → Claude Plan 서브에이전트
```

> 참고: revfactory/harness는 Claude의 *agent teams*(TeamCreate + SendMessage, 팀원 간 직접 통신)를 기본으로 둔다. 우리는 **Codex CLI 워커 + Claude 서브에이전트** 조합을 쓰며, 워커 간 직접 통신 대신 **merge를 Claude 메인이 책임지는** 단순한 모델을 택한다 (디버깅 용이, 토큰 절약, 깊이 ≤1 유지).

## 안티패턴

### A. Router 페르소나 (meta-orchestrator)

"어떤 에이전트를 부를지 결정하는 것"만 하는 페르소나.

**왜 실패하는가:** 도메인 가치 없는 순수 라우팅 계층. 패러프레이즈 홉이 늘어 정보 손실 + 토큰 비용 ~2배. 사용자는 이미 무엇을 원하는지 안다.

**대신:** 슬래시 커맨드를 추가/정제하고, 의도→커맨드 매핑을 `AGENTS.md`/`CLAUDE.md`에 문서화한다.

### B. 에이전트가 다른 에이전트를 호출

워커가 내부에서 또 다른 워커를 부르는 것.

**왜 실패하는가:** 전달되는 요약이 맥락을 잃고, 실패 모드가 곱해지고, 비용이 사용자에게 가려진다.

**대신:** 호출 측 워커가 보고서에서 후속 작업을 *권장*하고, 사용자/슬래시 커맨드가 두 번째 패스를 실행한다. (우리 맥락: Codex가 직접 다른 에이전트를 부르지 않는다.)

### C. 패러프레이즈하는 순차 오케스트레이터

사용자 대신 `/feature` → `/quality-gate` → `/commit`을 자동 호출하는 에이전트.

**왜 실패하는가:** 잘못된 방향을 조기에 잡는 사람 체크포인트를 잃고, 핸드오프마다 드리프트가 누적되며, 토큰 비용이 두 배가 된다.

**대신:** 사용자를 오케스트레이터로 유지. 권장 순서를 문서화하고 사용자가 호출하게 한다.

### D. 깊은 에이전트 트리

조정자가 또 다른 조정자를 부르고, 그게 또 워커를 부르는 구조.

**왜 실패하는가:** 계층마다 결정 가치 없이 지연·토큰만 추가. 디버깅이 다단계 조사가 되고, 리프 워커는 여러 요약 단계로 맥락을 잃는다.

**대신:** 조정 깊이를 **최대 1**로 유지한다 (슬래시 커맨드 → 워커들). merge는 메인 에이전트에서.

## 결정 흐름

```
하나의 산출물에 대한 하나의 관점인가?
├── 예  → 직접 호출. 끝.
└── 아니오 → 같은 구성이 반복되는가?
           ├── 아니오 → 즉석 직접 호출. 끝.
           └── 예 → 하위 작업이 독립적인가?
                   ├── 아니오 → 사용자 주도 순차 슬래시 커맨드 (패턴 3).
                   └── 예 → 병렬 fan-out + merge (패턴 2).
                           위 검증 체크리스트로 검증.
                           하나라도 실패 → 직접 호출로 폴백.
```
