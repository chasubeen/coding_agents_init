# Spec-Driven Development (스펙 주도 개발)

## 개요

코드를 한 줄 쓰기 전에 **구조화된 명세(spec)** 를 먼저 작성한다. 스펙은 나(Claude)와 사람 엔지니어 사이의 **공유된 단일 진실 공급원(source of truth)** 이다 — 무엇을 만드는지, 왜 만드는지, 무엇으로 "완료"를 판단하는지를 정의한다. 스펙 없는 코드는 추측이다.

이 프로젝트에서 스펙은 일회성 문서가 아니라 저장소 자체에 커밋되는 산출물이다 (`SPEC.md` 또는 `harness/` 하위). 저장소를 system of record로 삼는 Harness Engineering 원칙과 같은 결을 따른다 — 세션이 끊겨도 스펙만으로 작업을 이어갈 수 있어야 한다.

## 언제 쓰나

- 새 프로젝트나 기능을 시작할 때
- 요구사항이 모호하거나 불완전할 때
- 변경이 여러 파일·모듈에 걸칠 때
- 아키텍처 결정을 앞두고 있을 때
- 구현에 30분 이상 걸릴 작업일 때
- ML/DL 연구: 실험 설계(데이터셋, 평가 지표, 베이스라인, 성공 기준)가 코드보다 먼저 합의되어야 할 때

**언제 쓰지 않나 (When NOT to use):**

- 한 줄 수정, 오타 교정
- 요구사항이 자명하고 자기완결적인 변경
- 읽기·요약 같은 비변경 작업

## 핵심 프로세스 — 게이트형 워크플로

스펙 주도 개발은 네 단계로 이뤄진다. 현재 단계가 검증되기 전에는 다음 단계로 넘어가지 않는다.

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 사람 검토   사람 검토  사람 검토   사람 검토
```

각 게이트의 "사람 검토"는 이 프로젝트의 Human in the Loop 원칙과 직결된다. 고위험·비자명 결정은 사람 승인 없이 다음 단계로 진행하지 않는다.

### 1단계: SPECIFY — 명세

높은 수준의 비전에서 출발해, 요구사항이 구체화될 때까지 사람에게 명확화 질문을 던진다.

**가정을 즉시 드러내라.** 스펙 내용을 쓰기 전에, 내가 무엇을 가정하고 있는지 먼저 나열한다:

```
내가 하고 있는 가정:
1. 웹 애플리케이션이다 (네이티브 모바일 아님)
2. 학습 데이터는 이미 전처리된 상태로 주어진다
3. 평가 지표는 macro-F1 (accuracy 아님)
4. 단일 GPU 환경을 타깃으로 한다
→ 지금 정정해 주세요. 아니면 이 가정대로 진행합니다.
```

모호한 요구사항을 조용히 채워 넣지 마라. 스펙의 존재 이유는 코드가 쓰이기 *전에* 오해를 드러내는 것이다 — 가정은 가장 위험한 형태의 오해다.

**스펙 문서는 아래 6개 핵심 영역을 다룬다:**

1. **목표 (Objective)** — 무엇을 왜 만드는가? 사용자는 누구인가? 성공은 어떤 모습인가?

2. **명령어 (Commands)** — 도구 이름이 아니라 플래그까지 포함한 실행 가능한 전체 명령.
   ```bash
   Build: pip install -e .
   Test:  pytest -q --cov=src
   Lint:  ruff check . && mypy src
   Train: python main.py --config configs/default.yaml --seed 42
   ```

3. **프로젝트 구조 (Project Structure)** — 소스·테스트·문서가 각각 어디에 사는지.
   ```
   src/           → 애플리케이션/모델 소스
   src/models/    → 모델 정의
   src/utils/     → 공용 유틸리티
   tests/         → 단위·통합 테스트
   configs/       → 실험 설정
   ```

4. **코드 스타일 (Code Style)** — 세 문단의 설명보다 실제 코드 스니펫 한 개가 낫다. 네이밍 규칙, 포매팅, 좋은 출력의 예시를 포함한다.

5. **테스트 전략 (Testing Strategy)** — 어떤 프레임워크, 테스트 위치, 커버리지 기대치, 어떤 관심사를 어떤 레벨로 검증할지. 상세 패턴은 `test-driven-development` 스킬과 `harness/references/testing-patterns.md`를 따른다.

6. **경계 (Boundaries)** — 3단 체계. 이 프로젝트의 고위험 작업 등급표·거버넌스와 매핑된다:
   - **항상 한다 (Always):** 커밋 전 테스트 실행, 네이밍 규칙 준수, 입력 검증
   - **먼저 묻는다 (Ask first):** 스키마/데이터 파이프라인 변경, 의존성 추가, 설정 변경
   - **절대 안 한다 (Never):** 시크릿 커밋, vendor 디렉토리 편집, 실패하는 테스트를 승인 없이 삭제

**스펙 템플릿:**

```markdown
# Spec: [프로젝트/기능 이름]

## 목표 (Objective)
[무엇을 왜 만드는가. 사용자 스토리 또는 수락 기준.]

## 기술 스택 (Tech Stack)
[프레임워크, 언어, 핵심 의존성과 버전]

## 명령어 (Commands)
[Build, Test, Lint, Train/Run — 전체 명령]

## 프로젝트 구조 (Project Structure)
[디렉토리 레이아웃과 설명]

## 코드 스타일 (Code Style)
[예시 스니펫 + 핵심 규칙]

## 테스트 전략 (Testing Strategy)
[프레임워크, 테스트 위치, 커버리지 요구, 테스트 레벨]

## 경계 (Boundaries)
- 항상: [...]
- 먼저 묻기: [...]
- 절대 안 함: [...]

## 성공 기준 (Success Criteria)
[완료를 판단할 구체적·검증가능한 조건]

## 중단 조건 (Stop Conditions)
[언제 멈추고 재계획·보고할지. 예: 같은 에러 2~3회 반복 / scope drift / 되돌리기 어려운 작업 직전 승인 없음 / 가정이 틀림]

## 미지수 (Unknowns)
[시작 시점에 모르는 것. `unknown`으로 명시하고 해소되면 갱신. 추측으로 메우지 않는다]

## 미해결 질문 (Open Questions)
[사람 입력이 필요한 미결 사항]
```

> **중단 조건·미지수**는 하네스 고급 원칙이다 (`harness/references/harness-principles.md`). 미관찰 정보는 "없음"이 아니라 `unknown`으로 적는다 — "관찰되지 않음 ≠ 없음".

**지시를 성공 기준으로 재구성하라.** 모호한 요구사항이 들어오면 구체적·검증가능한 조건으로 번역한다:

```
요구사항: "모델 성능을 높여줘"

재구성된 성공 기준:
- 검증셋 macro-F1 ≥ 0.85 (현재 베이스라인 0.78)
- 추론 지연 단일 샘플 < 50ms (CPU)
- 동일 시드에서 재현 가능 (run-to-run 편차 < 0.5%p)
→ 이 타깃이 맞습니까?
```

이렇게 하면 "성능을 높인다"가 무엇인지 추측하는 대신, 명확한 목표를 향해 반복·재시도·문제해결을 할 수 있다.

### 2단계: PLAN — 계획

검증된 스펙으로 기술 구현 계획을 세운다:

1. 주요 컴포넌트와 그 의존성을 식별
2. 구현 순서 결정 (무엇을 먼저 만들어야 하는가)
3. 리스크와 완화 전략 기록
4. 병렬 가능한 것과 순차여야 하는 것 구분
5. 단계 사이 검증 체크포인트 정의

계획은 사람이 읽고 "맞아, 그 방향이 옳다" 혹은 "아니, X를 바꿔라"라고 말할 수 있도록 리뷰 가능해야 한다. 이 프로젝트에서 계획은 `harness/plan.md`에 적는다 (planner/builder/reviewer 프로토콜의 기준 문서).

### 3단계: TASKS — 작업 분해

계획을 이산적이고 구현 가능한 작업으로 쪼갠다:

- 각 작업은 집중된 한 세션에 완료 가능해야 함
- 각 작업에 명시적 수락 기준이 있어야 함
- 각 작업에 검증 단계(테스트·빌드·수동 확인)가 있어야 함
- 작업은 체감 중요도가 아니라 의존성 순으로 정렬
- 한 작업이 ~5개 이상의 파일을 건드리지 않게

작업은 `tasks/todo.md`(현재 세션 계획·진행·결과)에 체크리스트로 적고, 기능 단위 추적은 `harness/feature_list.json`에 매핑한다 (한 번에 하나만 `in_progress`, `passing` 전환은 검증+증거 필수).

**작업 템플릿:**
```markdown
- [ ] Task: [설명]
  - Acceptance: [완료 시 참이어야 할 것]
  - Verify: [확인 방법 — 테스트 명령, 빌드, 수동 확인]
  - Files: [건드릴 파일]
```

### 4단계: IMPLEMENT — 구현

작업을 한 번에 하나씩 실행하되 **증분(incremental)** 방식과 **TDD**를 따른다 — `test-driven-development` 스킬을 참조한다. 각 작업마다 RED → GREEN → REFACTOR를 거치고, 완료 선언 전 `/verify`로 정적·런타임·시스템 검증을 통과시킨다. 비자명한 구현 결정은 `doubt-driven-development`로 자체 적대적 검증을 거치고, 더 강한 검증이 필요하면 `/cross-check`로 Codex 교차검증을 제안한다.

## 패턴

### 스펙을 살아 있게 유지하기

스펙은 일회성 산출물이 아니라 살아 있는 문서다:

- **결정이 바뀌면 갱신** — 데이터 모델을 바꿔야 한다는 걸 발견하면, 스펙을 먼저 고치고 그다음 구현한다.
- **범위가 바뀌면 갱신** — 추가·삭제된 기능은 스펙에 반영한다.
- **스펙을 커밋** — 스펙은 코드와 함께 버전 관리에 들어간다 (`SPEC.md` 또는 `harness/`).
- **PR에서 스펙을 참조** — 각 PR이 구현하는 스펙 섹션으로 역링크한다.

### 산출물 연결 지점

| 단계 | 산출물 위치 |
|------|-----------|
| SPECIFY | `SPEC.md` 또는 `harness/` 하위 (커밋) |
| PLAN | `harness/plan.md` |
| TASKS | `tasks/todo.md` + `harness/feature_list.json` |
| IMPLEMENT | 구현 노트 `harness/implementation-notes.md`, 결과 `tasks/todo.md` |

## Common Rationalizations

| 변명 | 반박 |
|---|---|
| "이건 간단해서 스펙이 필요 없어" | 간단한 작업에 *긴* 스펙은 필요 없지만, 수락 기준은 여전히 필요하다. 두 줄짜리 스펙이면 충분하다. |
| "코드 다 짜고 나서 스펙 쓸게" | 그건 명세가 아니라 사후 문서다. 스펙의 가치는 코드 *전에* 명확성을 강제하는 데 있다. |
| "스펙이 우리를 느리게 만들어" | 15분짜리 스펙이 몇 시간의 재작업을 막는다. 15분 워터폴이 15시간 디버깅보다 낫다. |
| "어차피 요구사항은 바뀔 거야" | 그래서 스펙이 살아 있는 문서인 것이다. 낡은 스펙도 스펙 없는 것보다는 낫다. |
| "사용자가 뭘 원하는지 알아" | 명확한 요청에도 암묵적 가정이 숨어 있다. 스펙이 그 가정을 드러낸다. |

## Red Flags

- 어떤 문서화된 요구사항도 없이 코드를 쓰기 시작함
- "완료"의 정의를 명확히 하기 전에 "그냥 만들기 시작할까요?"라고 물음
- 어떤 스펙·작업 목록에도 없는 기능을 구현함
- 아키텍처 결정을 문서화 없이 내림
- "만들 게 뻔하다"는 이유로 스펙을 건너뜀

## Verification

구현으로 넘어가기 전에 증거 기반으로 확인한다:

- [ ] 스펙이 6개 핵심 영역을 모두 다룬다
- [ ] 사람이 스펙을 검토·승인했다
- [ ] 성공 기준이 구체적이고 검증 가능하다
- [ ] 경계(항상/먼저 묻기/절대 안 함)가 정의되어 있다
- [ ] 스펙이 저장소의 파일(`SPEC.md` 또는 `harness/`)로 저장·커밋되었다
- [ ] 작업이 `tasks/todo.md`에 분해되고 `harness/feature_list.json`에 매핑되었다
