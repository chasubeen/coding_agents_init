# Doubt-Driven Development (의심 주도 개발)

## 개요

자신만만한 답이 곧 정확한 답은 아니다. 긴 세션은 문맥을 쌓아가며, 아무도 눈치채지 못한 사이 가정을 조용히 "사실"로 둔갑시킨다. 의심 주도 개발은, 어떤 비자명한 출력이 확정되기 전에 **신선한 문맥의 검토자** — 승인이 아니라 **반증**에 편향된 — 를 실체화하는 규율이다.

이것은 사후 리뷰가 아니다. 사후 리뷰(`/code-review`, `/cross-check`)는 완성된 산출물에 대한 *판결*이다. 이 스킬은 *작업 중(in-flight)* 의 자세다 — 비자명한 결정을, 궤도 수정이 아직 싼 동안 교차심문한다.

> 이 프로젝트와의 연결: 의심 주도 개발은 우리 `/cross-check`(Codex 교차검증)와 **철학이 동일하다** — "자신만만함을 결론으로 두지 않고, 독립적·적대적 검증으로 증거를 더 모은다." 차이는 단계다. 의심 주도는 *내 추론을 내가 신선한 문맥으로 자체 적대 검증*하는 단계이고, `/cross-check`는 *다른 모델(Codex)에 교차모델 검증을 외주*하는 단계다. **스스로의 적대적 검증으로도 풀리지 않거나, 더 강한(교차모델) 검증이 필요하면 `/cross-check`로 Codex에 에스컬레이션**한다.

## 언제 쓰나

어떤 결정은, 아래 중 하나라도 참이면 **비자명(non-trivial)** 하다:

- 분기 로직을 도입·수정한다
- 모듈·서비스 경계를 가로지른다
- 타입 시스템이나 컴파일러가 검증할 수 없는 속성을 주장한다 (스레드 안전성, 멱등성, 순서, 불변식)
- 정확성이 미래의 독자가 볼 수 없는 문맥에 의존한다
- 영향 반경이 되돌릴 수 없다 (프로덕션 배포, 데이터 마이그레이션, 공개 API 변경)
- ML/DL: 데이터 누수·split 오염·재현성·지표 해석처럼 틀리면 결과 신뢰성 자체가 무너지는 영역

적용 시점:

- 불확실성 속에서 아키텍처 결정을 내리려 할 때
- 비자명한 코드를 커밋하려 할 때
- 비자명한 사실을 주장하려 할 때 ("이건 안전하다", "이건 확장된다", "이건 스펙과 일치한다")
- 완전히 이해하지 못한 코드에서 작업할 때

**언제 쓰지 않나 (When NOT to use):**

- 기계적 작업 (리네이밍, 포매팅, 파일 이동)
- 명확하고 모호하지 않은 사용자 지시를 따르는 것
- 기존 코드를 읽거나 요약하는 것
- 정확성이 자명한 한 줄 변경
- 순수 도구 작업 (테스트 실행, 파일 나열)
- 사용자가 검증보다 속도를 명시적으로 요구한 경우

키 입력마다 의심하면 아무것도 출하하지 못한다. 이 스킬은 위에 정의한 비자명한 결정에만 적용된다.

## 핵심 프로세스 — DOUBT 루프

스킬을 적용할 때 이 체크리스트를 복사해서 쓴다:

```
의심 사이클:
- [ ] 1단계 CLAIM    — 주장 + 왜 중요한지를 적었다
- [ ] 2단계 EXTRACT  — 아티팩트 + 계약을 분리하고, 내 추론은 떼어냈다
- [ ] 3단계 DOUBT    — 적대적 프롬프트로 신선한 문맥의 검토자를 호출했다
- [ ] 4단계 RECONCILE — 모든 finding을 아티팩트 텍스트에 대조해 분류했다
- [ ] 5단계 STOP     — 정지 조건을 만족했다 (사소한 finding / 3사이클 / 사용자 오버라이드)
```

### 1단계: CLAIM — 무엇이 확정되려 하는가

결정을 두세 줄로 명명한다:

```
CLAIM: "새 캐싱 레이어는 스펙에 기술된 read-heavy 워크로드에서
        스레드 안전하다."
왜 중요한가: 여기서 경쟁 조건이 생기면 사용자 데이터가 손상되고
            QA에서 잡기 어렵다.
```

이 정도로 압축해 적지 못한다면, 그것은 결정이 아니라 "느낌"이다. 검증하기 전에 먼저 표면화하라.

### 2단계: EXTRACT — 가장 작은 검토 단위

신선한 문맥의 검토자에게는 **아티팩트**와 **계약(contract)** 이 필요하다. 여정(reasoning)은 필요 없다.

- 코드: diff 또는 함수 — 파일 전체가 아니라
- 결정: 제안을 3~5문장 + 그것이 만족해야 할 제약
- 주장: 주장 + 그것을 뒷받침한다는 증거

**내 추론을 떼어내라.** 결론을 건네면 결론에 대한 검증이 돌아온다. 단위는 검토자가 한 번에 머릿속에 담을 수 있을 만큼 작아야 한다 — 500줄 PR이라면 먼저 분해하라.

### 3단계: DOUBT — 신선한 문맥의 검토자 호출

검토자의 프롬프트는 **반드시 적대적**이어야 한다. 프레이밍이 답을 결정한다.

```
적대적 검토. 이 아티팩트의 무엇이 잘못됐는지 찾아라.
저자가 과신하고 있다고 가정하라. 다음을 노려라:
- 명시되지 않은 가정
- 처리되지 않은 엣지 케이스
- 숨은 결합이나 공유 상태
- 계약이 깨질 수 있는 경로
- 이 변경이 깨뜨릴 수 있는 기존 관례
- 예상치 못한 입력에서의 실패 모드

검증하지 마라. 요약하지 마라. 이슈를 찾거나, 철저히 살핀 뒤에도
찾을 수 없다면 그렇다고 명시하라.

ARTIFACT: <아티팩트 붙여넣기>
CONTRACT: <계약 붙여넣기>
```

**ARTIFACT + CONTRACT만 전달하라. CLAIM은 전달하지 마라.** 검토자에게 내 결론을 건네면 동의 쪽으로 편향된다. 검토자는 아티팩트가 계약을 만족하는지 *독립적으로* 판단해야 한다.

이 프로젝트에는 `harness/agents/`에 역할별 리뷰어(reviewer, code-reviewer)가 정의되어 있다. 이들은 신선한 문맥으로 시작하므로 여기서 서브에이전트로 호출할 수 있다. 단, 페르소나 기본 응답 형태(강점+약점 균형 verdict)보다 **위 적대적 프롬프트가 우선**이다 — 적대적 프롬프트를 그대로 붙여 넣어 페르소나 기본형을 덮어쓴다.

#### 교차모델 에스컬레이션 — `/cross-check` (★ 이 프로젝트의 핵심 연결)

단일 모델 검토자는 원저자와 사각지대를 공유한다 — 더 차갑고 아키텍처가 다른 모델이 그걸 잡는다. 이 프로젝트에서 그 역할은 **Codex(보조 에이전트)** 가 맡으며, `/cross-check` 스킬로 호출한다.

이 프로젝트는 **Claude 제안 → 사용자 승인(Human in the Loop)** 제어 모델을 따른다. Codex는 자동으로 돌지 않는다.

**대화형 세션에서는 항상 제안하라. 조용히 건너뛰지 마라.**

1. 3단계의 자체(단일 모델) 적대적 검토를 마친 뒤, RECONCILE 전에 멈추고 사용자에게 제안한다:

   > "자체 적대적 검토를 마쳤습니다. Codex 교차모델 검증(`/cross-check`)을 받아볼까요? 옵션: diff / staged / 특정 파일 / plan / 건너뛰기."

2. 사용자가 승인하면 `/cross-check`로 위임한다. `/cross-check`는 사전 점검(`command -v codex`)을 하고, **read-only 샌드박스**로 Codex를 실행하며(Codex는 읽고 비평만, 수정은 Claude의 몫), ARTIFACT + CONTRACT + 적대적 프롬프트만 전달한다. CLAIM·세션 문맥은 넘기지 않는다.

3. **외부 CLI는 사용자 명시 승인 없이 호출하지 않는다** — 이것은 안전성의 핵심 속성이다. Codex가 없거나 실패하면 실패를 명시하고(조용한 폴백 금지), 수동 검토/다른 도구/건너뛰기를 제안한다.

4. 사용자가 건너뛰면 출력에 명시한다 ("단일(자체) 검토 결과만으로 진행합니다"). 건너뛰는 것은 괜찮다. *조용히* 건너뛰는 것은 안 된다.

**비대화형 문맥**(CI, `/loop`, autoresearch 루프, 스케줄 실행): 교차모델은 **건너뛰되**, 그 사실을 출력에 **명시**한다 ("교차모델 생략: 비대화형 문맥"). 외부 CLI를 명시 승인 없이 호출하지 않는다.

### 4단계: RECONCILE — finding을 되접기

검토자(자체 또는 Codex)의 출력은 데이터이지 판결이 아니다. **여전히 내가 오케스트레이터다.** 각 finding을 분류하기 전에 아티팩트 텍스트를 다시 읽고 대조한다 — 검토자를 고무도장처럼 찍는 것은 무시하는 것과 같은 실패 모드다.

각 finding을 이 **우선순위 순서**로 분류한다 (먼저 매칭되는 분류가 이긴다):

1. **계약 오독** — 검토자가 내가 준 CONTRACT가 불명확·불완전해서 플래그한 것. 계약을 먼저 고치고 다음 사이클에 재분류.
2. **유효 + 조치 필요** — 아티팩트 변경이 필요한 실제 이슈. 고치고 다시 루프.
3. **유효한 trade-off** — 실제 이슈지만 고치는 비용이 받아들이는 비용보다 큼. trade-off를 명시해 사용자가 보게 함.
4. **노이즈(오탐)** — 검토자가 갖지 못한 문맥 하에서는 실제로는 맞는데 잘못 플래그한 것. 근거를 적고 넘어간다.

신선한 검토자는 문맥이 없어서 틀릴 수 있다. "신선하다"는 이유만으로 양보하지 마라. (이것은 `/cross-check`의 "Codex 결과를 무비판적으로 수용하지 않는다" 원칙과 동일하다.)

### 5단계: STOP — 재귀가 아니라 경계 있는 루프

다음일 때 멈춘다:

- 다음 반복이 사소하거나 이미 고려된 finding만 내놓을 때, **또는**
- 3사이클을 완료했을 때 (혼자 4번째를 갈지 말고 사용자에게 에스컬레이션), **또는**
- 사용자가 명시적으로 "출하"라고 말할 때

3사이클 후에도 검토자가 여전히 실질적 이슈를 내놓는다면, 아티팩트가 준비되지 않았을 수 있다. 사용자에게 표면화하라 — 해결되지 않은 3사이클은 계속 돌릴 이유가 아니라 아티팩트에 대한 정보다.

아티팩트가 너무 커서 3사이클이 "명백히 부족"하다면, 아티팩트가 큰 것이다 — 2단계로 돌아가 분해하라. 경계를 들어올리지 마라.

## 패턴

- 확정된 실제 버그는 발견 즉시 `tasks/lessons.md` 승격 후보로 기록한다 (`/learn` 연동).
- `test-driven-development`의 RED 단계(실패하는 테스트)는 *행동에 대한* 의심을 구체화한 것이다 — 실패하는 테스트가 곧 그 행동 주장에 대한 DOUBT 단계를 만족한다.
- 검토자가 실제 실패 모드를 드러내면 `bugfix` 워크플로로 들어가 국소화·수정한다.

## Common Rationalizations

| 변명 | 반박 |
|---|---|
| "나 확신해, 의심 단계 건너뛸게" | 새로운 문제에서 확신은 정확성과 상관이 약하다. 확신의 순간이 바로 사각지대가 숨는 곳이다. |
| "검토자 띄우는 건 비싸" | 프로덕션에서 잘못된 커밋을 디버깅하는 게 더 비싸다. 검사는 경계가 있고, 버그는 없다. |
| "검토자는 어차피 트집만 잡을 거야" | 범위를 안 잡았을 때만 그렇다. "계약 하에서 실패하게 만들 이슈"로 프롬프트를 제약하라. |
| "마지막에 `/code-review`로 의심하면 돼" | 그건 최종 게이트다. 의심 주도는 방향이 틀렸을 때 일찍, 수정이 쌀 때 잡는다. PR 시점엔 늦다. |
| "매 단계 의심하면 영영 못 출하해" | 스킬은 비자명한 결정에만 적용된다. "언제 쓰지 않나"를 다시 읽어라. |
| "검토자가 반대했으니 내가 틀렸어" | 검토자는 내 문맥이 없다 — 반대는 정보이지 판결이 아니다. 아티팩트를 다시 읽고, 분류한 뒤 결정하라. |
| "교차모델은 늘 더 낫다" | 교차모델(Codex)은 단일 모델이 공유하는 사각지대를 잡지만, 비용·도구 취약성을 더한다. 매 대화형 사이클마다 *제안*하되, 가치 판단은 사용자에게 맡겨라. |
| "사용자가 한 번 yes 했으니 계속 Codex 호출해도 돼" | 호출마다 별도 승인이다. 아티팩트·프롬프트·플래그가 매번 바뀐다. 매 실행 전 정확한 명령을 사용자에게 재확인하라. |

## Red Flags

- 한 줄 리네이밍·포매팅 변경에 신선한 문맥 검토자를 띄움
- 아티팩트 텍스트를 다시 읽지 않고 검토자 출력을 권위로 취급
- 사용자에게 에스컬레이션 없이 3사이클을 초과해 루프
- "이거 괜찮아?"로 묻고 "이슈를 찾아라"로 안 물음
- 시간 압박 하에 고위험 결정의 의심을 건너뜀
- 변경되지 않은 아티팩트에 다시 검토자를 띄움 (같은 finding이 나온다 — 시간 끌기)
