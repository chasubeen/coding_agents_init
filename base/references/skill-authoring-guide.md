# 스킬 작성 가이드 (Skill Authoring Guide)

이 저장소의 `.claude/skills/<name>/SKILL.md`를 새로 만들거나 고칠 때의 표준. 우리가 이식한 엔지니어링 스킬들이 따르는 규약을 정리한 것이다.

관련 스킬: /harness, (모든 `.claude/skills/*`)

## 0. 만들기 전에 — 재사용 우선

새 스킬을 만들기 전, 기존 스킬과 겹치는지 확인한다.

| 상황 | 조치 |
|------|------|
| 완전히 포함됨 | 신규 생성 금지, 기존 스킬에 연결 |
| 부분 포함 + 일반화 가능 | 기존 스킬을 확장 |
| 의도된 도메인 특화 | 신규 생성 |
| 완전히 다른 기능 | 신규 생성 |

## 1. Frontmatter

```yaml
---
name: <디렉토리명과 정확히 일치, 소문자-하이픈>
description: <3인칭. "무엇을 하는지" + "언제 쓰는지(Use when)" 트리거. 1024자 이내>
user-invocable: true
argument-hint: "<인자 힌트 — 없으면 생략>"
---
```

- **description이 유일한 트리거 메커니즘**이다. *무엇*과 *언제* 둘 다 담는다.
- 트리거하면 *안 되는* 경계 조건도 넣는다 ("When NOT to use").
- ❌ **워크플로 단계를 description에 요약하지 말 것** — 요약하면 에이전트가 본문 대신 요약을 따른다.
- Claude는 스킬 발동에 보수적이므로, 트리거 표현은 다소 적극적으로.

## 2. 본문 작성 3원칙

1. **Why-First** — "항상 X 써라" 같은 강압 규칙보다 *이유*를 설명한다. ("pdfplumber는 셀 경계를 인식해 구조화 데이터를 반환한다" → 엣지케이스에서도 올바른 판단 유도)
2. **일반화** — 특정 사례("Q4 매출 열")가 아니라 원리 수준으로 ("수치를 암시하는 키워드가 있으면 숫자 타입으로 변환").
3. **명령형 어조** — "~할 수 있습니다"가 아니라 "~한다". 스킬은 지시서다.

토큰 절약: 모든 문장이 비용을 정당화하는지 본다. Claude가 이미 아는 일반 지식은 빼고, 긴 설명 대신 구체 예시.

## 3. 권장 섹션 구조

```
# 제목
## 개요            — 무엇을, 왜
## 언제 쓰나        — When to use + When NOT to use
## 핵심 프로세스     — 단계별. 입력/출력 대조 예시
## 패턴            — 재사용 가능한 접근
## Common Rationalizations  — 변명 → 반박 2열 표  ★
## Red Flags       — 지금 멈추고 다시 생각해야 할 신호  ★
## Verification    — 증거 기반 exit 체크리스트 `- [ ]`  ★
```

★ 표시 3종이 우리 스킬의 **anti-rationalization 장치**다. "되는 것 같다"는 추측을 차단한다. 모든 작업형 스킬에 반드시 넣는다.

**Common Rationalizations 예시:**

| 변명 | 반박 |
|------|------|
| "간단하니까 스펙 필요 없어" | 간단한 작업도 *완료 조건*은 필요하다. 두 줄 스펙이면 된다. |
| "코드부터 짜고 나중에 검증" | 그건 검증이 아니라 사후 정당화다. |

**Verification 예시:**

```
- [ ] 테스트가 실제로 실행되어 통과했다 (출력 첨부)
- [ ] 변경이 범위를 벗어나지 않았다
- [ ] 저장소만으로 다음 세션이 이어갈 수 있다
```

## 4. Progressive Disclosure (큰 스킬)

- **도메인 분리** — `references/`에 finance.md/sales.md 식으로 나눠 필요한 것만 로드.
- **조건부 상세** — 단순 경로는 본문에, 드문 상세는 별도 파일 참조로.
- **300줄+ 파일** — 상단에 목차.
- SKILL.md는 진입점, 보조 파일은 필요할 때만 로드한다.

## 5. 스크립트 번들링 판단

테스트 중 아래 신호가 보이면 `scripts/`로 번들링한다:

- 여러 실행에서 같은 헬퍼 스크립트를 매번 다시 만든다
- 매번 같은 설치/셋업을 반복한다
- 같은 다단계 절차를 반복한다 → 표준 절차로 기술
- 같은 에러 + 같은 회피책 → "알려진 문제와 해결" 섹션으로

번들 스크립트는 반드시 실행 테스트를 거친다.

## 6. 스킬에 넣지 않을 것

- README/CHANGELOG/설치가이드 등 부가 문서
- 테스트 결과·반복 이력 등 메타 정보
- 사람용 사용설명서 (스킬은 에이전트용)
- Claude가 이미 아는 일반 지식

## 7. 테스트 — with-skill vs without-skill

스킬이 실제로 행동을 바꾸는지 확인한다.

1. **without**: 스킬 없이 대표 작업을 시켜 baseline 결과를 본다.
2. **with**: 스킬을 로드하고 같은 작업을 시킨다.
3. **비교**: 차이가 없으면 그 스킬은 토큰 낭비다 — 더 구체적/지시적으로 고친다.
4. **트리거 검증**: description만으로 의도한 상황에서 발동하고, 비슷하지만 아닌 상황에선 발동 안 하는가.

> 새 스킬을 검증하고 싶으면 `/harness audit`(스킬은 Instructions 서브시스템)과 이 가이드를 함께 본다.
